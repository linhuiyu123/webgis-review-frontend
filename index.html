WebGIS 开发概述

🔑 本章先知道这几个词
  • B/S 架构（浏览器/服务器）：用户只要浏览器就能用，对比 C/S（要装客户端软件）。
  • HTTP（超文本传输协议）：浏览器和服务器“对话”的规则——你发请求、它回响应。
  • WebGIS：把 GIS（地理信息系统，管理“带位置的数据”）搬到网页上。
  • 矢量 / 栅格：点线面是矢量（道路、地块），像素网格是栅格（卫星影像、高程）。
  • 云 GIS（IaaS/PaaS/SaaS）：把 GIS 的存储和计算放到云端，按需使用。

深入精讲 ①：B/S 架构 —— 网页应用的基本骨架
理解了它，后面所有“前端/后端/服务器/请求响应”才有落脚点。
是什么

B/S = Browser/Server（浏览器/服务器）。意思是：你电脑上只装一个浏览器，真正干活的程序和数据都在远处的服务器上。你在浏览器里点一下，浏览器就向服务器“要”内容，服务器把结果送回来显示。对比 C/S（Client/Server），那是要你先下载安装一个专门的客户端软件（像桌面版 QQ、ArcGIS）才能用。

为什么

因为网页应用更新太频繁。如果是 C/S，每次升级都要让每一台电脑重新装一遍客户端，成本高。B/S 把核心逻辑集中到服务器，升级只改服务器一处，用户刷新网页就是新版；而且浏览器人人都有、跨平台，不用安装、即点即用。代价是：交互的流畅度和访问本地文件的能力不如桌面客户端。

怎么用

开发时你把网页代码（HTML/CSS/JS）和后端程序放到服务器；用户在浏览器输入网址 → 浏览器发 HTTP 请求 → 服务器处理后发回 HTTP 响应（网页或数据）→ 浏览器渲染出来。WebGIS 就是一个 B/S 应用，只不过它处理的是“带位置的空间数据”。

考试爱考“请求/响应分几部分”，看一个真实例子就懂了。
是什么

HTTP（超文本传输协议）是浏览器和服务器之间约定好的“对话格式”。一次完整对话分两条消息：浏览器发的请求、服务器回的响应。两者结构对称，都分三部分。

为什么

双方是两台不同的机器，必须有统一格式才能互相听懂——第一行说“干什么/结果如何”，中间一堆“附加说明”（首部），最后是“真正的数据”。HTTP 还特意设计成无状态：服务器默认不记得你上次来过，这样它简单、能扛海量用户；要“记住你”（如登录），就靠 Cookie。

怎么用

平时你不用手写，浏览器和后端框架自动生成。但要会看懂：下面左边是请求、右边是响应。请求第一行的 GET 是方法（GET 取数据、POST 提交、PUT 改、DELETE 删），/items?limit=10 是要的资源。

# 请求
GET /items?limit=10 HTTP/1.1   ← 请求行：方法+资源+版本
Host: example.com              ← 请求头(若干行)
Accept: application/json         说明我想要的格式
                               ← 空行，头结束
(GET 没有请求体)
# 响应
HTTP/1.1 200 OK                ← 状态行：版本+状态码+描述
Content-Type: application/json ← 响应头(若干行)
Content-Length: 81
                               ← 空行，头结束
{"items":[...], "count":10}    ← 响应体：真正的数据
状态码记三个：200=成功、404=资源不存在、500=服务器内部错误。方法记 PUT 幂等：同一个 PUT 调 1 次和 100 次结果一样；POST 不是（可能每次新建一条）。

2. HTTP 协议

HTTP 基本性质 简单、可扩展（HTTP headers）、无状态但有会话（本身无状态，可用 Cookie 实现有状态会话）。HTTP 需要可靠传输层协议，一般是 TCP。
TCP 三次握手

一次 HTTP 会话三阶段：建立连接（TCP三次握手）→ 客户端发请求并等待 → 服务器处理并响应。请求与响应都分三部分：

请求响应
第一行请求方法 + 参数 + 协议版本状态行（协议版本 + 状态码 + 描述）
中间HTTP 首部（多行，空行结束）HTTP 首部（空行结束）
末尾可选数据块（常用于 POST）可选数据块（响应数据）
小白读法：先把 HTTP 当成“点餐流程”

浏览器不是直接打开服务器硬盘上的文件，也不是直接查数据库。它做的事更像点餐：你告诉服务员“我要哪道菜、有什么要求”，服务员把请求交给后厨，后厨做完再把结果端回来。这里的“服务员”就是 Web 服务器，“后厨”可能是 GIS 服务器和空间数据库。

点餐类比HTTP / WebGIS 中对应什么你应该观察什么
菜单编号URL 路径，如 /layers/roads路径说明“要哪个资源”
点餐方式GET / POST / PUT / DELETE方法说明“查、增、改、删”
备注要求Query 参数或请求体参数说明“筛选条件或提交内容”
后厨出餐结果响应体：HTML、JSON、图片、瓦片浏览器最终拿到的是响应，不是数据库本身

3. WebGIS 介绍

定义WebGIS（万维网 GIS）是采用 Web 技术的 GIS，在满足 HTTP 和 TCP/IP 标准的 Internet 体系中实现地理信息的管理与处理。

与普通 Web 应用的区别：面向空间数据、需用空间数据库；数据量更大、结构更复杂；展示要考虑坐标系、空间分析等。

GIS 技术架构三阶段：独立主机结构（单机GIS）→ C/S 两层结构（局域网GIS）→ B/S 三层结构（WebGIS & 云GIS，将表示/业务/数据逻辑分给浏览器、应用服务器、数据服务器）。

WebGIS 三大组成：客户端（浏览器等）—— WebGIS 服务器（最重要）—— GIS 数据库（矢量/栅格等）。

4. 空间数据的存储与管理（高频考点）

【高频简答】空间数据存储管理的四个阶段
  1. 文件 + 关系数据库：几何用图形文件、属性用 DBMS，靠 ID 关联。成熟常用，但几何数据安全性/一致性弱，空间数据分开存储易破坏完整性。
  2. 全关系型空间数据库：几何和属性都用关系数据库管理（关系模型方式 / 二进制块方式）。统一存储，但几何解码慢、空间分析能力弱。
  3. 对象-关系数据库：在关系库上扩展，预定义空间数据类型与 API（如 PostGIS）。主流方法，速度快、支持多用户与事务。
  4. NoSQL 数据库：多样存储模式，适合非结构化空间数据。列存(HBase)、文档(MongoDB)、键值(Redis)、图(Neo4J)、对象(Ceph)。

5. 云 GIS

云计算三种服务模式（披萨比喻） IaaS 基础设施即服务（如 AWS、华为云）；PaaS 平台即服务（如 GAE、GEE）；SaaS 软件即服务（如在线软件、ArcGIS Online）。掌控程度 IaaS > PaaS > SaaS。

云计算五特征：随需应变自助服务、广泛网络访问、资源池化、弹性使用、可度量服务。

容器 vs 虚拟机：容器无自己的内核、直接运行于宿主内核，无硬件虚拟 → 启动秒级、占用 MB 级、性能接近原生、单机可上千。CNCF 代表技术：容器、服务网格、微服务、不可变基础设施、声明式 API（Docker、Kubernetes）。

深入精讲：盒模型 —— 网页排版的底层逻辑
网页布局看着乱，其实就是“摆盒子”。这一个概念吃透，CSS 就通了一半。
是什么

浏览器把每一个元素都当成一个矩形“盒子”。这个盒子由内到外有四层：content（内容，文字图片本身）→ padding（内边距，内容和边框之间的留白）→ border（边框，盒子的边线）→ margin（外边距，这个盒子和别的盒子之间的距离）。

为什么

因为网页要排版，就得控制“每个东西多大、离别人多远”。盒模型把“撑大自己内部空隙(padding)”和“推开别人(margin)”分成两件事，于是你能精确地调间距、对齐、留白。记两个易混点：padding 让盒子变大、margin 让盒子离别人远；padding 不能为负、margin 可以为负。

怎么用

用 CSS 给元素设这四层；想让块级盒子水平居中，给它定宽度再 margin: 0 auto。值的简写顺序是上右下左（顺时针）。

.card {
  width: 200px;            /* content 宽 */
  padding: 16px;          /* 四周内边距 */
  border: 2px solid red;  /* 边框=粗细 样式 颜色 */
  margin: 12px auto;      /* 上下12px，左右auto→水平居中 */
}

深入精讲：异步执行 —— 单线程的 JS 为什么不会被卡死
这是 JS 最难也最常考的点。先纠正一个常见错觉：异步不是“开了个新线程”。
是什么

JS 引擎本身只有一条线程（同一时刻只能干一件事）。“异步”指：遇到耗时操作（定时器、网络请求、读文件）时，JS 不在原地干等，而是把这件事交给浏览器提供的 Web API 去后台处理，自己立刻往下执行后面的代码；等那件事完成，它的回调函数被放进任务队列排队，等主线程空了再由事件循环取出来执行。

为什么

因为浏览器是单线程跑 JS 的：如果遇到“等服务器返回数据”这种要等几秒的事就傻等，那这几秒里页面点不动、划不动，体验极差。异步让主线程“把慢活外包出去、自己继续接客”，从而不阻塞页面。所以耗时操作都应异步做。

纠错：常说的“交给子线程”是不严谨的——真实模型是“交给浏览器的 Web API，完成后回调进入任务队列，由事件循环调度回主线程执行”。JS 主线程始终只有一条。
怎么用

6. 异步编程（高频简答）

同步 vs 异步JS 是单线程。同步：按顺序排队，前一任务结束才执行后一个，耗时任务会阻塞。异步：遇到异步任务先挂起交子线程，主线程继续往下，完成后再通知结果，不阻塞。
【高频简答】异步调用举三个例子，有何意义 三个例子：① 定时器（setTimeout/setInterval）；② 事件监听（addEventListener 回调）；③ 网络请求（Ajax / Axios 向服务器请求资源）（Promise 的成功/失败回调也算）。
意义：避免耗时操作阻塞主线程导致浏览器“卡死”失去响应，保持页面对其他事件的响应能力，提升用户体验——耗时操作都应异步执行。

🔑 本章先知道这几个词
  • OGC（开放地理空间联盟）：制定地图服务标准的国际组织，让不同 GIS 能互通。
  • WMS vs WFS：WMS 向服务器要“画好的地图图片”，WFS 要“矢量数据本身”（还能增删改）。
  • TMS / 瓦片：地图预先切成小图块，按需加载，又快又省流量。
  • GeoServer：开源地图服务器，把你的数据发布成 WMS/WFS 服务。
  • SLD：用 XML 描述“地图怎么画”（颜色、符号）的样式规则。
不认识的词随时查 📖 术语速查库
深入精讲：WMS / WFS / TMS —— 三种地图服务到底有啥区别
一句话先记牢：WMS 给你图片，WFS 给你数据，TMS 给你切好的小图块。
是什么

它们都是 OGC（开放地理空间联盟，定标准的国际组织）规定的“地图服务标准”，即服务器对外提供地图的几种标准接口。WMS=Web Map Service，要一张已经画好的地图图片；WFS=Web Feature Service，要矢量要素数据本身（坐标、属性，还能增删改）；TMS=Tile Map Service，把地图预先切成一堆小图块（瓦片）按需取。

为什么

因为不同场景需求不同：只想看地图→要图片最省事（WMS）；想查询/编辑数据（比如点中一栋楼改它的属性）→必须拿到数据本身（WFS）；地图又大又要频繁缩放平移→整张传太慢，于是预先切成小块、用到哪块传哪块（TMS），又快又省流量。

怎么用

这些服务一般由 GeoServer 这类地图服务器发布，前端（Leaflet/Cesium）按 URL 调用。WMS 的核心操作：GetCapabilities（问“你有哪些图层”）、GetMap（要图）。WFS 的核心：GetFeature（查要素，像 SQL 的 SELECT）、Transaction（增删改）。TMS 的 URL 长这样：/{z}/{x}/{y}.png，z 是缩放级别。

是什么

地图框架是一堆写好的 JavaScript 代码库，帮你在网页上显示地图、加底图、画点线面、响应缩放平移。你只要调用它的函数，不用自己处理“瓦片怎么拼、鼠标拖动怎么算坐标”这些苦活。四大代表：Leaflet（轻量、二维）、OpenLayers（二维、最守 OGC 标准）、Mapbox GL（二维、颜值高）、Cesium（三维地球）。

为什么

因为“在网页上做一个能交互的地图”非常复杂：要管投影、瓦片加载、图层叠加、事件处理……自己写要几个月。框架把这些封装好，几行代码就能出一个地图。其中能不能画三维取决于一项技术——WebGL（让网页用 GPU 硬件加速画 3D）。Leaflet/OpenLayers 偏二维（默认走 Canvas/SVG，不以内置 WebGL 为主），Mapbox/Cesium 用 WebGL，所以 Cesium 能渲染整个三维地球。

深入精讲：三种坐标系 —— 同一个点的三种“说法”
坐标系是 Cesium（也是整个 GIS）最大的门槛，务必分清这三种。
是什么

同一个地点，可以用三套坐标来描述：①地理坐标系（Cartographic）——经度、纬度、高度，就是“东经120°北纬30°”那种；②笛卡尔三维坐标（Cartesian3）——以地球质心为原点的 X/Y/Z 三个数；③屏幕坐标（Cartesian2）——这个点画在屏幕上的像素位置 (x, y)。

为什么

因为不同环节“说话方式”不同：人类习惯说经纬度；但计算机在三维空间里做旋转、定位、算距离，用X/Y/Z 三维直角坐标最方便；而最终要在平面屏幕上画出来，又得换成屏幕像素。所以同一个点要在三套坐标间换来换去。⚠ 关键纠错：Cartesian3 是“地心三维坐标”，不是地图学里把球面压平的“投影坐标系”（那是 GeoPandas 算面积用的，单位米的平面坐标），别混。还有：Cartographic 默认单位是弧度，不是度。

怎么用

记住几个转换函数即可（角度→必须先转）：

// 角度经纬度 → 三维笛卡尔（最常用，给实体定位）
Cesium.Cartesian3.fromDegrees(120, 30, 100);
// 角度 ↔ 弧度
Cesium.Math.toRadians(90);   Cesium.Math.toDegrees(1.57);
// 三维笛卡尔 → 弧度经纬度（拾取点位时用）
Cesium.Cartographic.fromCartesian(cartesian);

是什么

Shapefile 组成 必须文件：.shp（几何）、.shx（几何索引）、.dbf（属性）。可选：.prj（坐标参考系，WKT 描述）、.sbn/.sbx 等。

4. Primitive API（高频对比）

Primitive 由两部分组成：几何实例 GeometryInstance（定义结构）+ 外观 Appearance（定义着色，含 GLSL 顶点/片段着色器、渲染状态）。一个 Primitive 可有多个几何实例、但只能有一个外观。

【高频简答】Entity 与 Primitive 的异同 相同：都能绘制相同的几何图形（点/线/面/盒子等），Entity 底层实际调用 Primitive。
不同：

5. 材质与 Property 机制

  • 组件 / JSX：可复用的界面积木 / 在 JS 里直接写类 HTML 的语法。
  • state（状态）：组件内部会变的数据，必须用 setState/setter 改，不能直接赋值。

深入精讲：React 的核心闭环 —— 改 state，界面自动更新
React 的一切都围绕这一句话转。
是什么

React 是构建网页界面的框架，核心思想：你不直接去“手动改网页”，而是维护一份“状态数据”（state），并告诉 React“界面长什么样取决于这份数据”。一旦数据变了，React 自动帮你把界面更新成新样子。中间它用虚拟 DOM（真实网页的轻量副本）先算出“哪里变了”，只改变化的部分。

为什么

因为传统做法（用 JS 手动 document.getElementById 一处处改 DOM）在复杂页面里极其繁琐易错，而且频繁直接操作真实 DOM 很慢。React 让你只管“数据应该是什么”，把“怎么高效更新界面”交给框架；虚拟 DOM 的 diff 保证只动真正变化的那一小块，又快又省心。

两个铁律(高频考点)：① state 必须用 setter 改（setCount(1)），不能count=1 直接赋值，否则界面不更新；② props（父传子的数据）子组件只读不可改。
怎么用 
import { useState } from 'react';
function Counter() {
  const [count, setCount] = useState(0);   // 定义状态
  return <button onClick={() => setCount(count+1)}>{count}</button>;
  // 点击→setCount改状态→React自动重渲染按钮上的数字
}

小白读法：React 不是“新的 HTML”，而是“数据驱动界面”

React 的核心不是 JSX 长得像 HTML，而是你不再直接命令 DOM “把这里改成什么”，而是维护 state，让界面由 state 计算出来。点击按钮时，事件处理函数调用 setter 改 state，React 重新执行组件函数，生成新的虚拟 DOM，再只更新真实 DOM 里变化的地方。

深入精讲：地理坐标系 vs 投影坐标系 —— 为什么算面积前要“换坐标”
这是空间数据处理最容易踩的坑，务必弄懂。
是什么

坐标参考系（CRS）告诉数据“在地球上哪、用什么单位”。两大类：地理坐标系用经纬度（度）表示位置，基于球面，如 WGS84（EPSG:4326）；投影坐标系把球面压平成平面、单位是米，如 Web 墨卡托（EPSG:3857）、UTM。EPSG 就是给每种坐标系编的号。

为什么

因为经纬度的“1 度”不是固定长度——在赤道附近 1 度约 111 公里，到高纬度越来越短，而且经纬度是球面角度，直接拿来算平面面积/距离会严重失真。投影坐标系把地球摊平、单位统一成米，才能正确量长度算面积。所以：定位、显示用地理坐标系没问题；一旦要算面积、长度、缓冲区，先转成投影坐标系。

是什么

深入精讲：栅格数据模型 —— 一张“带地理坐标的数字表格”
看懂“像元矩阵 + 仿射变换”，栅格处理就不抽象了。
是什么

栅格数据就是一个行列排列的数字矩阵，每个格子叫像元(pixel)，存一个值（高程、温度、亮度…）。读进 Python 后，它本质就是一个 Numpy 二维数组。但光有数字不够，还要知道“第 0 行第 0 列这个格子对应地球上哪个坐标、每格代表多大”——这靠仿射变换(transform) 和 坐标系(crs) 来记录。

为什么

因为连续变化、难以用点线面描述的现象（影像、地形、气温场）用“一格一格的值”表示最自然。而仿射变换之所以重要：数组只有“行号列号”，没有地理位置；仿射变换就是一组系数，把“第几行第几列”换算成“真实经纬度/米坐标”。所以写栅格时如果丢了 crs 和 transform，这张图就“不知道自己在地球哪、每格多大”，没法和别的数据叠加分析。

怎么用

用 Rasterio 读写，把波段读成 Numpy 数组来处理（这叫地图代数，本质就是数组运算）。

67简答·第4章简述瓦片地图（TMS）的作用和原理。
参考答案
作用：高效快速加载渲染海量地图数据，减轻网络传输和客户端 CPU/GPU 渲染压力。原理：在服务端将矢量/栅格数据预先渲染好，按比例尺分级切割成图块（瓦片），客户端按需请求对应瓦片显示；统一投影到 Web 墨卡托使地图成正方形、避免极点投影到无穷远；链接形如 /{z}/{x}/{y}.png，第 z 级每方向 2ᶻ 块，一般 256×256 PNG。

69简答·第9章什么是虚拟 DOM？React 为什么快？
参考答案
虚拟 DOM 是 React 把代码先转换成的 JavaScript 对象（真实 DOM 的轻量映射）。更新时 React 比较虚拟 DOM 差异，通过微操作只对真实 DOM 做局部更新，避免整页重绘；同时屏蔽了跨浏览器差异。配合单向数据流，使渲染高效可预测。
70简答·第12章为什么计算面积/长度前要把地理坐标系转为投影坐标系？
参考答案
地理坐标系基于球面、以经纬度（角度）表示位置，受地球曲率影响，直接用经纬度算的距离/面积不准确，只适合定位；投影坐标系把球面投影到平面、单位为米，才能正确计算长度和面积。故需先 to_crs 到投影坐标系（如 EPSG:3857 或 UTM）。

AI：时空大数据挖掘与智能分析；GIS 智能体/时空大模型（如伏羲万象）辅助科研决策；自然语言交互降低使用门槛；遥感影像深度学习（分类、变化检测）。VR/三维：数字孪生、沉浸式三维可视化；推动二维地图→三维地图→虚拟现实的发展趋势；增强城市/灾害等场景的真实感与交互。结合：云原生、实时 GIS、智慧城市、物联网海量实时数据等背景下，AI 提供“智能分析”、VR 提供“沉浸表达”，共同推动 WebGIS 从功能向智能化、沉浸化演进。
